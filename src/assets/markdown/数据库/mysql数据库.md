MySQL 是关系型数据库管理系统。数据主要以表、行和列组织，通过 SQL 定义结构、查询数据和执行事务。高可用、读写分离与分库分表通常由复制、代理、中间件和应用架构共同完成，并不是一句“数据库原生支持”就能概括。

## SQL 与 MySQL 架构

SQL（Structured Query Language）用于定义、查询和操作关系数据。MySQL 服务器处理一条查询时，通常会经过连接与权限检查、解析、优化和执行等阶段，再由存储引擎访问数据。

MySQL 支持插件式存储引擎。InnoDB 是当前默认引擎，提供事务、行级锁、崩溃恢复和外键等能力。旧版 MySQL 曾有服务器查询缓存，但该功能已在 MySQL 8.0 移除，不能再把它列为现代执行链路的固定步骤。

常见字段类型包括数值、字符串、二进制、日期时间和 JSON。选择类型时应同时考虑取值范围、精度、排序规则、存储空间与查询方式。

## 事务与 ACID

事务是一组需要作为一个逻辑单元处理的操作。ACID 表示：

- **原子性（Atomicity）**：事务中的操作要么全部提交，要么在失败时回滚。
- **一致性（Consistency）**：事务把数据库从一个满足约束的状态带到另一个满足约束的状态；它也依赖正确的业务逻辑和约束设计。
- **隔离性（Isolation）**：并发事务的相互影响受到控制，效果取决于隔离级别和具体读写方式。
- **持久性（Durability）**：提交结果在规定的故障模型下能够保留。

### 并发异常

- **脏读**：读取了另一个尚未提交事务的修改，而该事务随后可能回滚。
- **不可重复读**：同一事务两次读取同一行，期间其他事务提交了更新或删除，结果不同。
- **幻读**：同一事务重复执行范围条件查询，期间其他事务提交了满足条件的新行，结果集合发生变化。
- **丢失更新**：并发修改覆盖了另一事务的结果，常见于“读—计算—写”没有并发控制的场景。

### SQL 隔离级别

| 隔离级别 | 标准语义概览 |
| --- | --- |
| Read Uncommitted | 可能读到未提交数据 |
| Read Committed | 每个一致性读读取该时刻已提交的数据 |
| Repeatable Read | 同一事务中的一致性读通常基于同一快照 |
| Serializable | 以最强隔离限制并发效果，通常降低并发度 |

InnoDB 默认隔离级别是 Repeatable Read。隔离级别并不单独“保证全部 ACID”，Serializable 也不是“完全服从 ACID”的别名。InnoDB 还会根据语句类型使用 MVCC、一致性读、记录锁、间隙锁和 next-key 锁，实际行为需要结合隔离级别与是否为锁定读判断。

## 并发控制

- **共享锁（S）**允许并发读取，但会限制冲突写入。
- **排他锁（X）**用于独占修改相关记录。
- 锁粒度可以涉及表、记录、间隙等；“行级锁”也可能因查询条件与索引范围影响多条记录。
- **MVCC** 保存可见版本信息，让许多一致性读无需阻塞写入。

不要把“加了事务”视为自动解决并发问题。更新库存、余额等逻辑仍需选择条件更新、锁定读、乐观版本号或其他合适方案。

## InnoDB 索引

InnoDB 常用 B+ 树组织聚簇索引和二级索引：

- 树的高扇出可以减少查找需要访问的页面数。
- 叶子页按键顺序链接，适合范围扫描与排序访问。
- 聚簇索引叶子记录保存整行数据；二级索引叶子记录保存索引列和主键值，查询其他列时可能需要回表。
- 哈希结构擅长等值定位，但通常不直接支持有序范围扫描，因此不能完全替代 B+ 树。

### 何时考虑索引

- 经常用于选择性较好的过滤、连接条件。
- 需要唯一性约束，例如业务编号。
- 可服务常见的排序、分组或覆盖查询。
- 表规模和查询频率足以抵消维护索引的成本。

低基数字段并不意味着索引一定无效，小表或频繁更新字段也不是绝对不能建索引。优化器会综合统计信息、回表成本和预计扫描行数选择执行计划。

### 常见的未使用索引原因

- B+ 树普通索引通常无法直接服务前导通配符查询，如 `LIKE '%text'`。
- 对索引列进行函数、计算或不合适的隐式类型转换，可能使条件无法形成可用的索引范围。
- 联合索引能否使用取决于索引列顺序、条件类型和查询目标；“最左前缀”是理解范围构造的起点，不是所有优化规则的终点。
- `OR` 各分支、数据分布或成本估算不合适时，优化器可能选择全表扫描，也可能使用 Index Merge 等访问方式。
- `NULL` 并不会天然让索引失效；是否声明 `NOT NULL` 应首先由数据语义决定。

应使用 `EXPLAIN` 或 `EXPLAIN ANALYZE` 检查真实执行计划，而不是只背“索引失效口诀”。

## 性能排查

1. 通过慢查询日志、性能模式和应用链路定位耗时位置。
2. 检查执行计划、扫描行数、回表、排序、临时表和锁等待。
3. 优化 SQL、索引与表结构，并用接近真实分布的数据验证。
4. 再考虑缓存、读写分离、分区或分片等架构方案。
5. 同时审视连接池、事务边界、磁盘、内存和网络，避免只靠增加硬件掩盖问题。
