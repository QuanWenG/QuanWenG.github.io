8. 并发事务带来的问题
- 脏读 一个事务对数据进行了修改，但其他事务读取了该事务的数据，但前一个事务进行了回滚
- 丢失修改 两个事务同时访问数据进行修改，前者的数据被丢失
- 不可重复读 一个事务没有结束时，另一个事务也进行读取，会因为第一个事务修改数据，导致第二个事务的读取结果不一致
- 幻读 一个事务读取的时候，其他事务插入数据，第一个事务的第二次读取会出现多出数据
- 不可重复读强调被修改 幻读强调查询记录增加

9. 并发事务的控制方式
- 锁
- - 共享锁（S锁，读锁）
- - 排他锁（X锁，写锁，独占锁）
- 根据颗粒度
- - 表级锁
- - 行级锁
- MVCC 多版本并发控制办法

10. sql定义了哪些事务隔离级别
- 读取未提交 能读取未提交的数据 
- 读取已提交 能读取已提交的数据 可防脏读
- 可重复读 能阻止不可重复读 可防脏读和不可重复读 mysql默认级别
- 可串行化 完全服从 ACID隔离级别 可防幻读脏读不可重复读

12. 为什么MySQL InnoDB 选择 B+ tree 作为索引的数据结构
- B+ tree vs B tree
- - 存储相同数据量级别下,B+ tree高度更低,IO次数更少
- - B+ tree 叶子节点使用双向链表,适合范围查询
- B+ tree vs 二叉树
- - B+ tree 叶子节点更多,树更矮,IO次数更少
- B+ tree vs Hash
- - B+ tree 支持范围查询,Hash不支持

13. 什么时候适用索引
- 字段有唯一性限制如商品编码
- 经常用于where查询条件的字段
- 经常用于group by  order by的字段

14. 什么时候不适用索引
- where group by order by中使用频率低的字段
- 字段中存在大量重复数据，不需要创建索引
- 表数据太少的时候，不需要创建索引
- 经常被更新的字段，不适合创建索引

15. 什么时候索引会失效
- 使用左或左右模糊匹配 "like %xx""like %xx%"
- 查询对索引列进行了计算 函数 类型转换操作
- 联合索引需要正确使用遵循最左匹配原则
- where子句中如果or前是索引列 or后不是索引列 那么索引失效
- 索引列最好设置NOT NULL 约束

16. 索引类型
- 数据结构分类 B+ tree索引 Hash索引 Full-Text索引
- 物理存储分类 聚簇索引（主键索引） 二级索引（辅助索引）
- 字段特性分类 主键索引 唯一索引 普通索引 前缀索引
- 字段个数分类 单列索引 联合索引
