Redis 是以内存数据结构为核心的数据存储系统，常用于缓存、计数、排行榜、限流、消息传递和协调任务。它可以通过 RDB、AOF 等机制持久化数据，但“以内存为主”不等于“永远不会落盘”或“宕机绝不丢数据”。

## Redis 为什么快

- 大多数操作直接访问内存，并使用针对不同数据结构优化的编码。
- 网络层使用非阻塞 I/O 与事件循环处理大量连接。
- 常用命令的执行路径短，协议解析相对简洁。
- 核心命令通常在主线程串行执行，减少了共享数据结构上的锁竞争。

性能仍取决于命令复杂度、数据大小、持久化配置、网络和服务器负载。对大集合执行高复杂度命令，或传输超大键值，都可能长时间阻塞主线程。

## 与本地缓存的区别

本地缓存访问链路更短，但每个应用实例持有独立副本，容量、淘汰和一致性管理都分散在各进程中。Redis 提供跨实例共享的数据视图、集中容量管理和更丰富的数据结构，但会引入网络延迟、外部依赖与运维成本。

两者并非互斥：可采用本地一级缓存加 Redis 二级缓存，但必须明确失效通知、过期时间、缓存穿透和一致性容忍范围。

## 数据结构与典型用途

- `String`：缓存对象、计数器、位图。
- `Hash`：字段型对象。
- `List`：有序序列和简单队列。
- `Set`：去重、集合运算。
- `Sorted Set`：排行榜、按分数调度。
- `Stream`：可持久化的消息流、消费组与待处理消息追踪。

使用 Redis 实现分布式锁、限流或队列时，需要把超时、故障转移、重复执行和业务幂等纳入设计，不能只依赖一条命令名称推断完整正确性。

## 消息传递

### Pub/Sub

发布者通过 `PUBLISH` 向频道发送消息，在线订阅者通过 `SUBSCRIBE` 接收。Redis Pub/Sub 使用 **at-most-once** 交付语义：消息发出后不会持久保存，订阅者断线或处理失败时消息可能永久丢失。

因此 Pub/Sub 适合在线通知和允许丢失的广播。如果需要消息留存、确认、消费组或故障恢复，应考虑 Redis Streams 或专门的消息系统。

### Streams

Redis Streams 以追加日志形式保存条目，支持按 ID 读取、阻塞读取和消费组。消费组可以记录待确认消息，但这并不自动等于“业务恰好执行一次”：消费者仍需确认消息、处理重试，并让业务操作具备幂等性。

## 延时任务

常见实现包括：

1. 把执行时间作为 Sorted Set 的分数，工作进程取出到期任务。
2. 使用具备延时队列封装的客户端库，例如 Redisson。
3. 在可靠性要求更高时使用原生支持延时或重试语义的消息系统。

键过期通知不适合作为可靠任务队列：通知依赖配置，可能延迟，订阅者离线时也可能丢失。任何延时方案都需要处理多实例竞争、任务领取后的崩溃、重复投递和幂等执行。Redisson 等封装能减少实现工作，但不能无条件保证“绝不丢失、绝不重复”。

## 线程与 I/O 模型

把 Redis 简称为“单线程”并不完整：

- Redis 的核心命令执行通常由主线程串行完成。
- 服务器还会使用后台线程或子进程处理释放内存、持久化等工作。
- Redis 4.0 引入了部分异步删除能力，避免大对象释放长时间阻塞主线程。
- Redis 6.0 起可配置 I/O 线程协助网络读写；命令执行仍主要保持串行语义。

这种模型让常规命令避免大量锁与线程切换，但 CPU 密集命令仍会阻塞其他请求。扩展吞吐时通常结合命令优化、流水线、分片或集群，而不是假设增加单实例工作线程就能线性加速。

## 使用注意事项

- 给缓存设置符合业务容忍度的过期与淘汰策略。
- 防范缓存穿透、击穿和雪崩，但不要套用固定模板忽略业务分布。
- 避免生产环境使用会扫描巨大键空间或处理大集合的阻塞命令。
- 根据可接受的数据丢失窗口配置 RDB、AOF、复制和故障转移。
- 监控延迟、内存碎片、热键、大键、复制积压与持久化状态。
