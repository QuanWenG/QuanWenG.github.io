网络分层把复杂通信拆成相对独立的职责。分层是理解模型，不代表每个协议都能毫无争议地归入唯一层级；工程实现中也常出现跨层优化。

## OSI 与 TCP/IP 模型

| TCP/IP 模型 | 大致对应的 OSI 层 | 主要职责 | 常见协议或技术 |
| --- | --- | --- | --- |
| 应用层 | 应用层、表示层、会话层（5～7） | 面向应用的数据交换 | HTTP、DNS、SMTP、SSH |
| 传输层 | 传输层（4） | 端到端传输与端口复用 | TCP、UDP、QUIC |
| 网际层 | 网络层（3） | 跨网络寻址与路由 | IP、ICMP |
| 链路层 | 数据链路层、物理层（1～2） | 本地链路传输 | Ethernet、Wi-Fi、ARP |

这种对应关系是便于理解的近似映射，并非协议实现的硬边界。分层让协议可以独立演进、复用下层能力，并把故障定位限制在更清晰的范围内。

## 常见协议

### 应用层

- **HTTP**：传输超文本及通用 Web 资源。
- **DNS**：域名系统，把名称解析为地址或其他记录。
- **SMTP**：邮件发送与转交；POP3、IMAP 常用于邮件访问。
- **FTP**：传统文件传输协议。
- **SSH**：加密的远程登录与通道协议。
- **RTP**：承载实时音视频数据，常与其他控制或协商协议配合。

### 传输层与网际层

- **TCP** 面向连接，提供可靠、有序的字节流，并实现流量控制与拥塞控制。
- **UDP** 面向数据报，不保证送达、顺序或去重，应用可以在其上建立自己的可靠性机制。
- **IP** 负责跨网络寻址与尽力而为地转发数据报。
- **ICMP** 传递错误报告和诊断信息。
- **ARP** 在 IPv4 局域网中解析 IP 地址对应的链路层地址，通常被放在链路层或网络层边界讨论。
- **OSPF、RIP、BGP** 用于交换路由信息。
- **NAT** 是地址转换机制，不是一种与 IP、ICMP 并列的端到端传输协议。

## 从输入 URL 到页面展示

一次导航通常经历以下阶段，具体步骤会因缓存、HTTP 版本和连接复用而变化：

1. 解析 URL，检查浏览器缓存、HSTS 与 Service Worker 等本地状态。
2. 通过 DNS 获取目标地址；结果可能来自浏览器、操作系统、递归解析器或缓存。
3. 建立传输连接：HTTPS 常涉及 TCP 与 TLS，HTTP/3 则基于 QUIC。
4. 发送 HTTP 请求，服务器或中间缓存返回响应。
5. 浏览器解析 HTML，获取 CSS、JavaScript、字体和图片等子资源。
6. 构建 DOM 与 CSSOM，执行脚本，完成布局、绘制与合成。

这些步骤并非始终严格串行。浏览器会预加载、并行请求并复用连接。

## HTTP 与 HTTPS

HTTPS 是运行在 TLS 安全通道上的 HTTP。TLS 主要提供服务器身份认证、传输加密和完整性保护；它不能保证服务器业务可信，也不能阻止用户主动提交敏感数据给恶意站点。

HTTP 的默认端口通常是 80，HTTPS 通常是 443，但端口可以显式改变。HTTPS 会增加握手与加密工作，不过会话恢复、硬件加速和现代协议已显著降低额外成本，不能简单概括为“HTTP 快、HTTPS 慢”。

## GET 与 POST

- `GET` 请求目标资源的表示，语义上应当是安全且幂等的；参数经常放在 URL 查询字符串中，但这不是协议强制的唯一数据表达方式。
- `POST` 向目标资源提交数据，可能创建资源、触发处理或产生其他副作用；它在通用语义上既不安全也不幂等。
- 请求方法不决定是否加密。敏感信息应通过 HTTPS 传输，并避免出现在容易进入历史记录、日志和 Referer 的 URL 中。
- 权限校验、CSRF 防护和输入验证都不能只依赖选择 GET 或 POST。

## WebSocket

WebSocket 为客户端和服务器提供持久的全双工消息通道。经典 HTTP/1.1 场景下，客户端先发送 Upgrade 握手，服务器接受后返回 `101 Switching Protocols`，随后双方在同一连接上传输 WebSocket 帧。关闭时通常交换关闭帧，再结束底层连接。

HTTP 并不是“单向协议”，而是客户端请求、服务器响应的消息交换模型；WebSocket 的关键差异是连接建立后双方都能独立主动发送消息，且每条消息无需重复携带完整 HTTP 头。

### 与轮询比较

- **短轮询**：客户端定期请求。实现简单、兼容性好，但实时性和空请求成本取决于轮询间隔。
- **长轮询**：服务器暂时保持请求，直到有数据或超时；响应后客户端再发起下一次请求。
- **WebSocket**：适合双方都需要低延迟、频繁发送消息的场景，但需要处理连接保活、断线重连、代理兼容和扩缩容。

## Server-Sent Events（SSE）

SSE 通过 HTTP 响应以 `text/event-stream` 格式持续向客户端发送文本事件。浏览器 `EventSource` API 原生支持事件 ID 和自动重连，适合通知、日志与文本流式响应。

SSE 的标准数据方向是服务器到客户端；客户端仍可用普通 HTTP 请求向服务器发送数据。WebSocket 则提供单连接上的全双工通信，并支持文本与二进制消息。选择时应依据数据方向、代理环境、消息格式和连接规模，而不是只比较“谁更实时”。
